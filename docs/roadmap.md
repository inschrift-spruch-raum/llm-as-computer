# transturing 路线图（长期方案）

本文档基于当前工作树中的现有代码、测试与设计文档整理，目标是为 `transturing` 补齐当前实现与 `misc/Can LLMs Be Computers.md` 所描述理论之间的主要缺口，形成一份可长期执行的研发路线图。

本文档不是对未来结果的承诺，而是对当前已知缺口、依赖关系、验证标准和风险控制方式的规划。所有阶段性工作都应继续遵守仓库现有原则：

- 统一内部 ISA 与 `Trace` / `TraceStep` 协议
- 维护 NumPy 与 Torch 两后端的语义一致性
- 在扩展 WASM 支持时保持边界清晰且测试先行
- 只写当前代码和测试可以支撑的事实

## 1. 当前状态与核心缺口

从当前仓库可见代码来看，项目已经实现了以下基础能力：

- 统一的内部栈机 ISA
- 以 `Trace` 为中心的执行结果表示
- NumPy 参考解释器语义
- Torch 路径中的解析式 embedding / attention / compiled weights
- 受限 WASM32 `i32` 子集的二进制摄入、结构化控制流 lowering 与链接
- 围绕 NumPy/Torch parity 的测试体系

但如果目标是逐步靠近理论文档中的“可作为计算机的 transformer 执行系统”，当前仍存在几类关键缺口：

1. **Torch 路径仍由 Python 外层执行循环驱动。** 当前 `TorchExecutor.execute()` 逐步调用单步前向，而不是在模型内部完成长程 rollout。
2. **注意力检索仍是线性扫描。** 当前 parabolic key 编码已经存在，但尚未引入凸包类加速结构，因此尚不具备理论中强调的快速解码路径。
3. **部分运算语义仍在 Python 中直接求值。** 这意味着 Torch 路径尚未把全部执行逻辑都收束到统一的“模型内可组合部件”中。
4. **WASM 支持范围仍较窄。** 当前明确仅支持受限 `i32` 子集，与“更一般的 C/WASM 输入”之间还有明显距离。
5. **缺少面向长程执行与性能的系统化验证。** 当前测试重心是语义正确性与 parity，而不是复杂度、吞吐、极长轨迹稳定性。
6. **缺少围绕输出 token、程序编译进权重、训练/混合架构的上层实验框架。** 这些内容在理论文档中出现，但当前仓库尚未进入该阶段。

本路线图的总体思路不是一次性追求所有理论目标，而是先补齐“正确性 → 结构纯化 → 性能路径 → 输入范围 → 更高层研究能力”这一依赖链。

## 2. 总体目标

长期目标可分为五个层级：

### 2.1 保持当前系统的可验证核心

在任何阶段，都不能牺牲以下已有优势：

- NumPy 后端作为参考语义
- Torch 后端作为模型化语义
- `TraceStep(op, arg, sp, top)` 作为跨后端对比协议
- `tests/test_consolidated.py` 与 `tests/test_wasm_binary.py` 提供的回归基线

### 2.2 把 Torch 路径从“模型辅助执行器”推进到“更纯的模型内执行器”

这里的重点不是先追求速度，而是减少外层 Python 控制与 Python 语义分支，让更多执行规则以统一的模型结构、权重编译和状态表示来表达。

### 2.3 在不破坏语义的前提下，引入真正的快速检索路径

当前项目已经具备 parabolic addressing 的数学基础。下一步应围绕它建立明确的数据结构与验证框架，而不是直接把性能优化和架构重写混在一起。

### 2.4 扩展可摄入的程序范围

受限 WASM 子集是当前稳定边界，但长期目标应是在测试充分的前提下，逐步向更现实的编译产物靠近。

### 2.5 为更上层研究方向预留演进空间

包括但不限于：

- 输出 token / 流式外显 trace
- 程序到权重的更强编译路径
- 训练或混合执行架构
- 更标准的 transformer 结构表达

这些方向应建立在前几层已稳定之后，而不是提前侵蚀底层语义清晰度。

## 3. 规划原则

### 3.1 先协议，后实现

若某项工作会改变：

- opcode 语义
- `TraceStep` 的字段含义
- 内部内存/地址编码方式
- WASM lowering 约定

则必须优先明确协议层改动，再分别落实到 NumPy、Torch、测试与文档。

### 3.2 先 NumPy 语义，再 Torch 映射

对于执行逻辑的新增与修复，应优先在 NumPy 路径上确认“目标语义到底是什么”，再决定如何映射到 Torch 路径。这样可以避免把 Torch 路径中的实现技巧误当成规范本身。

### 3.3 先 parity，后性能

性能优化只能建立在严格的 parity 之上。任何显著改变 Torch 路径内部状态组织方式的工作，都必须先证明：

- 不破坏既有 trace 语义
- 能被现有回归测试覆盖
- 对新增行为有明确测试

### 3.4 分层推进，避免多变量同时改动

以下工作不应在同一个大改动里混做：

- 扩 ISA
- 改 Torch embedding 维度布局
- 改 WASM 支持范围
- 引入新性能数据结构
- 修改输出协议

路线图的每个阶段都应只有一个主要变量。

## 4. 阶段路线图

## 阶段 0：基线固化与可观测性补强

### 目标

在进入更激进的结构演进前，先把当前系统的正确性边界、性能基线与调试接口固定下来，避免后续阶段无法判断自己究竟是“真的进步了”，还是只是“改得更复杂了”。

### 建议工作

1. 增加一组长期保留的 benchmark / profiling 基线。
   - 覆盖短程序、中等程序、深循环、调用栈、局部变量、堆访问、WASM 编译路径。
   - 至少区分 NumPy 与 Torch 的运行时间、步数、内存占用增长趋势。
2. 为 Torch 路径补充更细的内部状态断言与调试钩子。
   - 例如记录 attention 命中位置、memory embedding 长度、非线性分支命中分布。
3. 将“长程执行失败模式”纳入测试与文档。
   - 比如步数过长时的速度退化、embedding 增长模式、极值/地址冲突行为。
4. 梳理并冻结当前 `D_MODEL = 51` 的维度分配说明。
   - 明确哪些维度是硬约束，哪些可以为后续阶段重构。

### 交付标准

- 新增 `benchmarks/` 或等效基线脚本目录
- 文档中明确当前复杂度与已知退化场景
- 能稳定复现实验结果

### 风险

- 若没有这一阶段，后续所有“优化”都难以证明自己是否真的有价值。

## 阶段 1：Torch 路径语义纯化

### 目标

减少 Torch 后端中对 Python 直接求值逻辑的依赖，使 Torch 路径更接近“以统一可组合部件表达执行语义”的目标。

### 当前主要问题

从现有代码可见，Torch 后端虽然已经用 attention 和解析式权重编码了大量状态读写，但仍保留了 `_compute_nonlinear()` 这类 Python 直接求值逻辑。这会导致：

- Torch 路径难以被描述为更纯粹的“模型内执行”
- 后续如果要做更大规模 rollout 或输出 token，结构上不够统一
- 某些行为更像“模型辅助解释器”，而不是“编译执行器”

### 建议工作

1. 按 opcode 家族拆分 Torch 路径中的 Python 语义逃逸点。
   - 算术/比较/位运算/一元运算分别建立追踪清单。
2. 为每一类逃逸点设计可替代的表达方式。
   - 可是额外 attention 头、线性路由层、专门的 token/embedding 结构，或更明确的 compiled submodule。
3. 在不改变外部 `Trace` 契约的前提下，逐步减少 `_compute_nonlinear()` 的职责。
4. 同步维护 NumPy/Torch parity 测试，确保每次纯化都可回归验证。

### 交付标准

- Torch 路径中 Python 直接求值逻辑显著减少，并有清单化说明
- 现有 consolidated tests 与 WASM tests 全部通过
- 新增针对“纯化后关键 opcode 家族”的专项 parity 测试

### 风险

- 若试图一口气消除全部 Python 逻辑，可能导致 Torch 路径结构失控。
- 若没有逐家族推进，回归定位会非常困难。

## 阶段 2：快速检索路径（Hull / Convex Hull 风格加速）

### 目标

在保持现有 parabolic addressing 语义不变的前提下，把当前线性扫描式检索替换为更接近理论目标的快速数据结构。

### 关键前提

当前项目已经具备以下条件：

- program/stack/local/heap/call stack 都使用同类抛物线 key 编码
- NumPy 与 Torch 两条路径都能复现同一检索语义

因此，这一阶段的核心不是“重新发明编码”，而是“证明加速结构与当前语义等价”。

### 建议工作

1. 抽象出统一的“地址型检索接口”。
   - 把当前 `argmax` 线性扫描背后的行为定义成清晰的接口契约。
2. 先在 NumPy 或纯 Python 环境实现一个可验证的 hull-based reference。
   - 目标是验证数据结构语义，而不是先追求极限速度。
3. 以属性测试或差分测试方式证明：
   - 新检索路径与旧扫描路径在所有已支持 memory space 上返回相同结果。
4. 再决定如何嵌入 Torch 路径。
   - 可以是 executor 级缓存、head 级加速层，或独立 memory index 组件。
5. 建立性能回归基线。
   - 至少展示长轨迹下复杂度曲线变化。

### 交付标准

- 存在可独立测试的快速检索模块
- 差分测试证明与旧路径语义一致
- 长轨迹 benchmark 显示复杂度改善

### 风险

- 若把“更快”放在“语义等价”之前，极易破坏 parity。
- 若没有统一接口，优化会渗透到太多模块，导致维护困难。

## 阶段 3：从单步前向到更长程的模型内 rollout

### 目标

将当前由 Python 驱动的“单步前向 + 外层循环”推进到更接近理论中的“模型内部连续执行”结构。

### 为什么这一阶段不应提前

如果在快速检索路径和 Torch 语义纯化之前就开始做 rollout，得到的只是一个更难调试、仍然很慢、且语义边界更模糊的系统。

### 建议工作

1. 先明确 rollout 的最小定义。
   - 是多步 batched forward。
   - 是显式内部状态张量滚动。
   - 还是离散 token trace 形式的自回归生成。
2. 引入中间过渡版本。
   - 例如“k 步窗口内 rollout”，而不是直接追求无限长程自回归。
3. 为 rollout 结构建立新的正确性约束。
   - 逐步 trace 是否与单步执行完全一致。
   - call/return、trap、branch target 是否在窗口边界保持一致。
4. 评估是否需要调整 `Trace` 或新增内部 rollout trace 结构。
   - 若要改动公开 trace 契约，应单独立项，不应偷偷修改。

### 交付标准

- 至少有一种“多步执行但保持语义一致”的 rollout 原型
- 与现有逐步执行方式在指定程序集上 trace 对齐
- 明确记录仍保留在 Python 层的控制职责

### 风险

- rollout 很容易把问题从“执行语义”转移成“序列同步 bug”。
- 若没有清晰最小定义，项目会陷入无限扩张。

## 阶段 4：扩展 WASM 支持范围

### 目标

在维持现有受控边界的前提下，让仓库能够摄入更现实的编译产物，从“受限演示输入”逐步走向“更一般的 C/WASM 输入”。

### 建议优先级

WASM 扩展不应以“支持越多越好”为目标，而应优先选择：

1. 能显著提升现实输入可达性的能力
2. 对内部 ISA 和双后端影响可控的能力
3. 能通过测试清楚界定支持/拒绝边界的能力

### 建议顺序

1. 更完整的函数签名与 locals 支持边界梳理
2. 更现实的内存访问形态
3. 导入/全局等显著改变模块链接语义的特性
4. 更高阶的 WASM 机制（如 table / indirect call）

### 每次扩展必须同步完成的事项

- `wasm_binary.py` 的解析与验证逻辑
- NumPy/Torch 两后端对生成程序的消费验证
- `tests/test_wasm_binary.py` 增补正例与反例
- README / design 文档边界更新

### 交付标准

- 每新增一种支持能力，都有正向测试与拒绝测试
- 不出现“悄悄放宽边界但没有文档和测试”的情况

### 风险

- WASM 范围扩展极易牵动调用约定、内存模型与控制流 lowering。
- 若没有阶段性冻结，输入面会迅速超过执行器本身的可验证能力。

## 阶段 5：输出协议与外显 trace

### 目标

为系统增加更接近理论文档的“对外输出”能力，使执行结果不仅体现在内部 `TraceStep.top`，也能以更明确的协议被外部消费。

### 建议工作

1. 明确“输出”在当前系统中的语义位置。
   - 是新 opcode。
   - 是新 trace 事件。
   - 还是独立的 output side-channel。
2. 评估它对现有 parity 规则的影响。
   - 当前 parity 聚焦 `TraceStep(op,arg,sp,top)`；引入输出后是否扩展协议，必须明确。
3. 为流式消费设计稳定接口。
   - 例如 trace iterator、output event stream 或 token 序列视图。

### 交付标准

- 输出能力被明确建模，而非只靠“读取最后一步栈顶”隐式表达
- 新协议不破坏现有调试与 parity 能力

### 风险

- 若输出语义设计不慎，会同时扰动 ISA、测试和所有现有示例。

## 阶段 6：更高层研究能力

### 目标

在底层执行系统稳定后，再进入理论文档中更长远的研究方向。

### 可能方向

1. **程序到权重的更强编译路径**
   - 当前权重更像固定解释器。长期可探索特定程序或程序片段直接编译为更专门化的权重结构。
2. **更标准的 transformer 结构表达**
   - 研究是否需要引入层结构、残差、标准 attention block，以便靠近理论中的更一般架构。
3. **训练或混合式架构**
   - 在不破坏已有解析式可验证核心的前提下，探索学习型组件与执行型组件的边界。
4. **面向外部系统集成的接口**
   - 如更稳健的 API 分层、编译缓存、运行时观测、实验框架。

### 交付标准

- 有独立实验边界，不直接污染核心执行协议
- 对每个方向都先写实验目标、成功标准与退出条件

### 风险

- 若在底层基础未稳时提前进入这一阶段，项目很容易从“有验证的运行时”退化成“概念堆叠”。

## 5. 关键依赖关系

从依赖关系看，推荐顺序如下：

1. **阶段 0：基线固化与可观测性补强**
2. **阶段 1：Torch 路径语义纯化**
3. **阶段 2：快速检索路径**
4. **阶段 3：更长程的模型内 rollout**
5. **阶段 4：扩展 WASM 支持范围**
6. **阶段 5：输出协议与外显 trace**
7. **阶段 6：更高层研究能力**

其中要特别强调：

- 阶段 2 不宜在阶段 0 之前开始，因为缺少性能与复杂度基线。
- 阶段 3 不宜在阶段 1、2 之前开始，因为会把“还不够纯、还不够快”的结构直接放大。
- 阶段 4 虽可与阶段 1 部分并行，但不宜在底层执行语义仍频繁变化时大规模扩展输入面。

## 6. 每个阶段的统一验收要求

无论进入哪个阶段，所有改动都建议至少满足：

1. `python -m pytest tests/test_consolidated.py tests/test_wasm_binary.py`
2. `python -m basedpyright src tests`
3. 必要时执行 `python -m ruff check .`
4. 对新增能力补充专项测试
5. 对文档边界进行同步更新

另外，凡是会影响 `TraceStep.op`、`TraceStep.arg`、`TraceStep.sp`、`TraceStep.top` 的改动，都应被视为高风险改动，必须显式说明：

- 这是 bug fix、协议修订，还是新能力扩展
- 为什么 parity 变化是合理的
- 哪些测试被更新，为什么

## 7. 建议的里程碑定义方式

为了让路线图能被真正执行，而不是停留在口号层，建议后续每个阶段都拆分成以下三层成果：

### 7.1 RFC / 设计说明

先写清楚：

- 目标
- 不做什么
- 涉及哪些模块
- 对协议、测试、文档的影响

### 7.2 参考实现

优先做最小可验证实现，而不是一上来追求完整高性能版本。

### 7.3 产品化收敛

当参考实现已证明语义正确，再考虑：

- 性能优化
- API 暴露方式
- 文档和示例
- 更广的输入覆盖

## 8. 近期最值得启动的事项

如果只从“下一步最值得做什么”出发，而不立刻展开大规模改造，最推荐的起点是：

1. **建立 benchmark 与 profiling 基线**
2. **把 Torch 中 Python 直接求值逻辑做清单化拆解**
3. **为快速检索路径写一个独立 RFC**

这样做的原因是：

- 成本相对可控
- 不会立刻破坏当前 parity 基线
- 能显著减少后续阶段的决策不确定性

## 9. 一句话总结

`transturing` 的长期演进，不应从“追求更炫的理论目标”开始，而应从**巩固可验证语义、纯化 Torch 路径、建立快速检索基础、谨慎扩展 WASM 输入面**开始，逐步把当前“有数学骨架的实验性运行时”推进为“更接近理论执行模型、又仍然能被测试和维护的系统”。
